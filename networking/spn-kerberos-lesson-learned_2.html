SPN & Kerberos Authentication

Lesson Learned & Resolution Guide

English / 中文   |   Active Directory Infrastructure

Overview / 概述

English

SPN (Service Principal Name) is a unique identifier stored in Active Directory. It links a service instance to the security account that runs it, and is mandatory for Kerberos authentication.

Missing, incorrect, or duplicate SPNs cause Kerberos to fail silently. The system automatically falls back to slower NTLM authentication, which results in sudden slowness when users access HTTP-based web applications.

中文

SPN（服务主体名称）是 Active Directory 中存储的唯一标识符，用于将服务实例与其运行的安全账户关联，是实现 Kerberos 身份验证的必要条件。

当 SPN 缺失、配置错误或重复时，Kerberos 验证会失败。系统会自动降级使用速度较慢的 NTLM 验证，这会导致所有用户访问 HTTP 网页服务时出现性能突然下降的问题。

Memory Map / 知识地图

Why Slowness Happens Suddenly / 为何性能突然变慢

English

  • Duplicate SPNs: Same SPN registered on two different accounts → Kerberos cannot resolve.
  • Account Changes: App pool identity, service password, or computer account reset.
  • DNS Updates: New CNAME alias created but no corresponding SPN added.
  • AD Replication: Changes take time to sync; issue appears after old tickets expire.
  • Policy Changes: Authentication order, encryption types, or firewall rules modified.
  • Time Drift: Time difference > 5 minutes between client and domain controller.

中文

  • SPN 重复：同一 SPN 被注册到两个不同账户，导致 Kerberos 无法解析。
  • 账户变更：应用池运行身份、服务账户密码或计算机账户被重置。
  • DNS 变更：新增域名别名（CNAME），但未配置对应的 SPN 记录。
  • AD 复制延迟：配置变更后需要时间同步，问题通常在旧验证票据过期后集中出现。
  • 策略变更：验证方式顺序、加密类型或防火墙规则被修改。
  • 时间不同步：客户端与域控制器时差超过 5 分钟，直接导致 Kerberos 失效。

Where & How to Register / 注册位置与规则

English

  • Domain user accounts: If a service (e.g. an IIS app pool) runs as a domain user, register the SPN on that user account, not the computer account.
  • Local System / Network Service: Register the SPN on the computer account instead, since the service runs under the machine's identity.
  • Cover every access path: Register the short hostname, the FQDN, and any CNAME/DNS aliases users might type in a browser — a missing alias SPN causes NTLM fallback for that alias only.
  • Check before adding: Always run a duplicate check across the domain first; adding a conflicting SPN breaks Kerberos for both accounts.

中文

  • 域用户账户：若服务（如 IIS 应用池）以域用户身份运行，SPN 应注册在该用户账户上，而非计算机账户。
  • Local System / Network Service：此时服务以机器身份运行，SPN 应注册在计算机账户上。
  • 覆盖所有访问路径：短主机名、完整域名（FQDN）以及用户可能使用的 CNAME 别名都需要各自注册 SPN，遗漏别名会导致该别名访问时降级为 NTLM。
  • 添加前先检查：务必先在全域范围内检查重复项，冲突的 SPN 会同时破坏两个账户的 Kerberos 验证。

Common commands / 常用命令

# List SPNs registered on an account / 查看账户已注册的 SPN setspn -L domain\serviceaccount # Check for duplicate SPNs domain-wide / 检查全域内重复的 SPN setspn -X # Register an SPN / 注册 SPN setspn -S HTTP/webapp.contoso.com domain\serviceaccount setspn -S HTTP/webapp domain\serviceaccount # Remove an SPN / 删除 SPN setspn -D HTTP/webapp.contoso.com domain\serviceaccount

Diagnosis Steps / 排查步骤

English

  1. Confirm the fallback: check the active auth package with klist or a Kerberos-filtered network capture.
  2. Run setspn -X to look for duplicate SPNs across the domain.
  3. Verify time sync between the client and domain controller with w32tm /stripchart /computer:<DC>.
  4. Confirm DNS resolution for every hostname/alias in use with nslookup.
  5. Review the Security event log on the client and server for Kerberos error IDs (4, 11, 17, 27).

中文

  1. 确认是否降级：使用 klist 或带 Kerberos 过滤的网络抓包，确认当前使用的验证方式。
  2. 执行 setspn -X，检查全域内是否存在重复的 SPN。
  3. 使用 w32tm /stripchart /computer:<DC> 验证客户端与域控制器的时间是否同步。
  4. 使用 nslookup 确认所有正在使用的主机名/别名解析是否正常。
  5. 检查客户端与服务器的安全事件日志，查看 Kerberos 相关错误事件 ID（4、11、17、27）。

Lessons Learned / 经验总结

English

  • Register SPNs as a required step when provisioning a new service account or DNS alias — not an optional cleanup task afterward.
  • Keep a central inventory of which account owns which SPN, so duplicate checks are fast during an incident.
  • Add an SPN/Kerberos sanity check to the change-management checklist for any AD account or DNS change.
  • Set up alerting on NTLM fallback so an SPN problem surfaces before users notice slowness.

中文

  • 在新建服务账户或 DNS 别名时，将 SPN 注册作为必须步骤，而非事后才补做的清理工作。
  • 维护一份集中的 SPN 归属清单，方便在故障发生时快速排查重复项。
  • 在涉及 AD 账户或 DNS 变更的变更管理清单中，加入 SPN/Kerberos 检查项。
  • 针对 NTLM 降级设置告警，以便在用户察觉变慢之前就发现 SPN 问题。
