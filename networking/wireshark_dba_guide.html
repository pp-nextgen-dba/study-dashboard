0. Beginner Basics 新手基础 start here

What is a packet? 什么是数据包？

Data travelling over a network is chopped into small chunks called packets. Each packet is wrapped in layers of "envelopes" — like a letter, inside an envelope, inside another envelope. Wireshark shows you every layer when you expand a packet in the Details pane.
网络中传输的数据会被切分成一个个小块，称为"数据包"。每个数据包像信封套信封一样，包裹着多层信息。展开Wireshark的数据包详情面板即可看到每一层。

Simplified layers 简化分层
LayerExample
Frame 帧Ethernet / Wi-Fi
Network 网络层IP address (who → who)
Transport 传输层TCP/UDP (port, reliability)
Application 应用层HTTP / TDS / pgsql / TNS
Client ↔ Server model 客户端-服务器模型

The client (browser, app, DB tool) sends a request; the server (web server, MSSQL/PostgreSQL/Oracle engine) sends back a response. Nearly every capture you'll read is one of these request/response pairs.
客户端（浏览器、应用、数据库工具）发送请求，服务器（Web服务器或数据库引擎）返回响应。几乎所有抓包分析都是在读这一组组请求与响应。

The TCP handshake, in plain terms TCP三次握手（通俗讲解）

Before any SQL query or HTTP request can be sent, TCP does a quick "hello" — 3 packets, no data yet. If you ever see a capture where the handshake never finishes, the connection never really started.
在发送任何SQL查询或HTTP请求之前，TCP会先完成一次简短的"握手"——3个数据包，此时还没有真正的数据。如果握手没有完成，说明连接根本没建立起来。

sequenceDiagram participant C as Client 客户端 participant S as Server 服务器 C->>S: SYN (let's connect 我要连接) S->>C: SYN-ACK (ok, you there? 好的，收到) C->>S: ACK (confirmed 确认) Note over C,S: Connection established 连接建立，可以开始发送数据/SQL/HTTP请求

Glossary 术语表

TermPlain-English meaning 通俗解释
Port 端口A number identifying which service on a machine (80=web, 1433=MSSQL, 5432=PostgreSQL, 1521=Oracle). Like an apartment number at a street address.
TCP vs UDPTCP = reliable, ordered, confirms delivery (used by DBs & HTTP). UDP = fire-and-forget, faster but no guarantee (used by DNS, SQL Browser discovery).
RTT (Round-Trip Time) 往返时延Time for a packet to go out and its acknowledgment to come back — a core "is it slow?" measurement.
Latency vs Throughput 延迟 vs 吞吐量Latency = how long one packet takes. Throughput = how much data moves per second. A link can have low throughput but still low latency, or vice versa.
Retransmission 重传Sender didn't get confirmation in time, so it resends the packet — usually means packet loss somewhere.
Socket 套接字The combination of IP address + port + protocol that uniquely identifies one connection.
Dissector 解析器The piece of Wireshark that knows how to read a specific protocol (e.g. the "tds" dissector understands MSSQL's language).
NAT (SNAT/DNAT) 网络地址转换A device (e.g. load balancer) rewrites source/destination IP so the client and real server never talk to each other's real address directly.
Beginner tip: You don't need to understand every layer to be useful — start by filtering to your protocol (http, tds, pgsql, tns) and reading the "Info" column in the Packet List. Everything else can wait.
新手提示：不需要一开始就搞懂每一层，先用协议过滤（http/tds/pgsql/tns），看数据包列表的"Info"信息列即可，其余内容可以慢慢学。

1. Core Concepts 核心概念

Capture filter — set before capture starts, BPF syntax (e.g. port 1433), reduces what's written to disk.
抓包过滤器：开始抓包前设置，BPF语法，减少写入磁盘的数据量。

Display filter — applied after capture, Wireshark syntax (e.g. tcp.port==1433), used 95% of the time.
显示过滤器：抓包后筛选已捕获数据，日常分析主要使用。

Tip: Capture broadly (e.g. just host <server_ip>), narrow down later with display filters.
建议：抓包范围宽松一些（例如只按IP过滤），之后用显示过滤器精细筛选。

5. Web Slowness & Middleboxes Load BalancerRiverbed

For "the website is slow" tickets, the path is rarely just Client → Server. A load balancer and/or a WAN optimizer usually sit in between, and each one changes what your capture looks like.
对于"网站慢"类工单，路径很少是客户端直连服务器，中间通常还有负载均衡器和/或广域网优化设备（如Riverbed），它们都会改变抓包的表现形式。

flowchart LR C[Client 客户端] -->|"TCP #1"| LB[Load Balancer
VIP + SNAT/DNAT] LB -->|"TCP #2"| RVL[Riverbed LAN-side
optional] RVL -.optimized WAN link.-> RVR[Riverbed WAN-side
optional] RVR -->|"TCP #3"| S[Backend Server 后端服务器]

注意：客户端到服务器实际上可能是多段独立的TCP连接（Client→LB, LB→Riverbed, Riverbed→Server），每一段都要单独抓包对比才能定位延迟发生在哪一段。

Why / Who / How — the plain-language framework 为什么 / 谁负责 / 怎么设计——简明框架

Simple analogy first: think of Riverbed as a courier relay between two warehouses. It repacks boxes to ship faster and cheaper between the two points — it never changes the delivery address, and it's invisible to the sender and receiver.
先打个比方：Riverbed就像两个仓库之间的"快递中转站"。它把包裹重新打包压缩以便更快更省地在两点之间运输——从不改变收货地址，对发件人和收件人来说都是"隐形"的。

WHY do we need it? 为什么需要它？
  • Physics: a round trip across a long-distance WAN (e.g. branch ↔ HQ, or cross-country/cross-country DC) has a minimum delay just from distance — no amount of bandwidth fixes that. 跨地域的一次往返本身就有最低延迟，带宽再大也无法消除。
  • Cost: WAN/MPLS bandwidth between sites is expensive; sending less data (compression/dedup) means the company can use cheaper links. 站点间专线带宽昂贵，压缩去重后可以用更便宜的链路。
  • Chatty protocols: some protocols need many small round trips to do one task; each one pays the full distance-delay, so they feel painfully slow over a WAN without help. 部分协议一次操作需要多次小的往返，每次都要付出完整的距离延迟。
WHO manages it? 谁来管理？
  • Network/Infrastructure team owns the appliances, licensing, and tunnel health — same team that owns routers/firewalls. 网络/基础设施团队负责设备、许可证与隧道健康状态，和路由器防火墙是同一团队。
  • App/DB teams don't configure it, but should know it exists on the path and loop in network team when diagnosing performance. 应用/数据库团队不负责配置，但应知道它的存在，排查性能问题时要拉上网络团队。
  • Security team gets involved only if Riverbed is configured to inspect/optimize encrypted (TLS) traffic, since that requires sharing certificates/keys. 只有当Riverbed需要对加密流量做深度优化时才涉及安全团队，因为需要共享证书/密钥。
HOW to design apps around it 应用设计时如何考虑
  • Ask early in design: "does this traffic cross a WAN with Riverbed in the path?" — involve network team before go-live, not after complaints. 设计初期就问网络团队路径上是否有Riverbed，而不是等用户投诉后才问。
  • Avoid chatty designs: batch multiple small calls into fewer, larger ones where possible — fewer round trips = less exposed to WAN distance-delay. 尽量把多次小请求合并成更少、更大的请求，减少往返次数。
  • Expect a "cold vs warm" curve: don't panic if the very first user/report load is slower — repeat access is expected to speed up. 预期"冷热"效应：第一次加载慢属正常，无需立即报警。
  • If traffic is end-to-end encrypted (HTTPS/TLS everywhere), Riverbed can still speed up the TCP mechanics but can't compress/dedupe the payload unless keys are shared — decide this trade-off with security/network up front. 若全程加密，Riverbed仍可优化TCP机制，但无法压缩/去重载荷内容，除非共享密钥，需提前与安全/网络团队权衡。
3-question framework to reuse for any new project 可复用的"三个问题"框架:
  1. Does this traffic cross a WAN link with Riverbed (or similar) in the path? 该流量是否会经过带有Riverbed的广域网链路？
  2. Is it encrypted end-to-end? 是否全程加密？ (affects how much optimization is possible / 影响可优化的程度)
  3. Is the protocol chatty (many small round trips per transaction)? 协议是否"多话"（一次操作需要多次往返）？ (affects how much warm-cache benefit to expect / 影响热缓存能带来多少提升)

Load Balancer (F5 / HAProxy / Nginx / ALB) LB

What changes in the capture 抓包中的变化Why it matters 为什么重要
Client sees the LB's VIP as "the server"; the real backend never appears in the client-side capture.You must capture on both sides (client-facing and server-facing) to see the whole picture. 需要在LB两侧分别抓包才能看到完整路径。
Server-side capture shows the LB's IP as source (SNAT), not the real client IP — unless X-Forwarded-For / proxy protocol is used.Don't be confused when "client IP" in the backend capture is actually the LB. 后端抓包里的"客户端IP"其实是负载均衡器的IP。
Health-check probes mixed in with real traffic (regular, tiny requests to a fixed path like /health).Filter these out so they don't skew your read of real user traffic. 过滤健康检查流量，避免误判为真实用户请求。

Useful filter: http.request.uri contains "/health" (exclude with !) to separate health checks from real traffic.

Riverbed / WAN Optimization (SteelHead) Riverbed

What changes in the capture 抓包中的变化Why it matters 为什么重要
Riverbed splits one logical connection into a LAN-side and a WAN-side TCP connection (transparent proxy).What looks like "the same connection" in your capture may actually be two separate TCP sessions stitched together. 看似同一条连接，实际可能是两段独立的TCP会话被拼接在一起。
TCP options may show a Riverbed auto-discovery probe (vendor-specific TCP option during the handshake).Confirms Riverbed is in the path even if no one told you. 可用于确认路径中确实存在Riverbed设备。
Data reduction (SDR/compression) changes byte counts and can make throughput look unusually good or introduce extra latency on cache misses.Don't assume "less data on the wire" always equals "faster" — first-pass (cold) transfers may be slower than expected. 首次传输（未命中缓存）时反而可能比预期慢。
Common misdiagnosis trap: Riverbed's connection splitting can look like retransmissions or resets to someone unfamiliar with it — always confirm with the network team whether WAN optimization is active on the path before concluding "packet loss."
常见误诊陷阱：Riverbed的连接拆分在不了解的人看来可能像是重传或连接重置——下结论说"丢包"之前，务必先向网络团队确认该路径是否启用了广域网优化。

Troubleshooting approach 排查思路

  1. Capture as close to the client as possible, and as close to the backend server as possible, at the same time.
    尽量在客户端附近和服务器附近同时抓包。
  2. Compare timestamps for the same logical request across both captures — the gap tells you which segment (client↔LB, LB↔Riverbed, Riverbed↔server) owns the delay.
    对比同一请求在两侧抓包中的时间戳，差值即为延迟所在的具体路段。
  3. Exclude health-check noise, then re-check retransmissions/ack_rtt per segment rather than end-to-end.
    排除健康检查流量后，分段核查重传与ack_rtt，而不是只看端到端总时延。

Detailed example: how Riverbed sits between Client → Web Server → DB 详细示例：Riverbed在客户端—Web服务器—数据库之间的作用

Typical scenario: a branch-office client hits a web application at HQ/datacenter over a slow WAN link, and that web server talks to a DB server that's usually right next to it on the same LAN (so Riverbed normally optimizes the Client↔Web Server leg, not the Web Server↔DB leg — unless the DB is itself at another site).
典型场景：分支机构的客户端通过较慢的广域网访问总部/数据中心的Web应用，该Web服务器通常与数据库服务器在同一局域网内（因此Riverbed通常优化的是"客户端↔Web服务器"这一段，而不是"Web服务器↔数据库"这一段——除非数据库本身也在异地）。

flowchart LR subgraph Branch["Branch Office 分支机构"] C[Client 客户端] end subgraph WAN["WAN Link 广域网链路"] SH1[Riverbed SteelHead
near-end 近端] SH2[Riverbed SteelHead
far-end 远端] end subgraph DC["Datacenter 数据中心 (LAN)"] W[Web Server] D[(DB Server)] end C -- "Leg 1: LAN, plain TCP
client thinks it talks to Web Server directly" --> SH1 SH1 -. "Leg 2: Optimized WAN tunnel
compression + SDR + MX-TCP" .-> SH2 SH2 -- "Leg 3: LAN, plain TCP" --> W W == "usually same LAN, no Riverbed needed" ==> D

Key point for the team: unlike a load balancer, Riverbed does not change IP addresses (no NAT). The client's packets still show the real web server IP as destination, and vice versa — SteelHead devices sit "in-path" and transparently intercept, so from the app's point of view it looks like one direct connection, even though underneath it is 3 separate TCP legs.
关键点：与负载均衡器不同，Riverbed不做地址转换（无NAT）。客户端数据包中的目的IP仍然是Web服务器的真实IP，反之亦然——SteelHead设备是"透明在线"部署，应用层看起来像一条直连，但底层实际上是3段独立的TCP连接。

How the connection gets split (auto-discovery) 连接如何被自动识别并拆分

sequenceDiagram participant C as Client 客户端 participant N as Near SteelHead 近端 participant F as Far SteelHead 远端 participant W as Web Server C->>N: SYN (carries Riverbed "probe" TCP option) N->>F: forwards SYN with probe query F->>W: SYN (normal, on behalf of client) W->>F: SYN-ACK F->>N: SYN-ACK (carries probe response = "I am here, let's optimize") N->>C: SYN-ACK (probe stripped, looks like a normal handshake) Note over C,W: Now 3 TCP legs exist:
Client-Near (LAN) | Near-Far (optimized WAN) | Far-WebServer (LAN) C->>N: HTTP/SQL request (plain) N-->>F: compressed / deduped over WAN tunnel F->>W: HTTP/SQL request (plain, reconstructed)

客户端发出的SYN包中带有Riverbed探测选项；近端SteelHead截获后转发探测请求给远端；远端确认存在Riverbed后，双方"握手"确认可以优化，随后的数据在两端SteelHead之间以压缩、去重、优化TCP（MX-TCP）的方式传输，到达远端后再还原成普通流量转发给真正的服务器。

First load vs repeat load — a giveaway pattern 首次加载 vs 重复加载——一个可辨识的规律

ScenarioWhat you'd see 现象What it suggests 说明
First time opening a report/page (cold) 首次打开（冷数据）Full data volume crosses the WAN tunnel; normal or even slightly slower than expected.SDR reference library hasn't seen this data yet — nothing to reduce. SDR尚未建立该数据的参考库，无法压缩去重。
Same report/page again shortly after (warm) 短时间内重复打开（热数据）Byte count on the WAN leg drops sharply; feels much faster.Riverbed replaced repeated byte patterns with small references — this is expected and good. 重复的数据模式被替换为小引用，属正常优化效果。
Every load is slow, warm or cold 无论冷热都慢No improvement on repeat access.Points away from Riverbed and toward the backend (DB query, web server processing) or the LAN leg. 说明问题更可能出在后端（数据库查询、Web服务器处理）或局域网段，而非Riverbed。
Talking-point script for the meeting: "If this page is slow only the first time and fast on reload, that's consistent with Riverbed's data reduction warming up — not a bug. If it's slow every single time regardless of reload, Riverbed likely isn't the cause and we should look at the DB query plan or web server processing time instead."
会议话术参考："如果这个页面只是第一次慢、刷新后就快，这符合Riverbed数据缩减预热的正常表现，不是故障。如果每次都慢，跟是否刷新无关，那问题更可能不在Riverbed，应该去看数据库执行计划或Web服务器处理时间。"

6. Application Caching & Redis Redis

Simple analogy first: instead of walking to the filing cabinet in the basement (the database) every single time you need a fact, you write it on a sticky note on your desk (the cache). Next time you need the same fact, you glance at the sticky note — much faster than another trip downstairs.
先打个比方：与其每次都跑去地下室的档案柜（数据库）翻找资料，不如把常用的答案写在桌上的便签纸（缓存）上。下次要用时看一眼便签就行，比再跑一趟地下室快得多。

Where caching sits in the path 缓存在整条链路中的位置

flowchart LR C[Client 客户端] --> W[Web / App Server] W -- "1: check cache first" --> R[(Redis
in-memory cache)] R -- "HIT: return cached value 命中" --> W R -. "MISS: not found 未命中" .-> D[(DB Server
Oracle/PostgreSQL/MSSQL)] D -- "2: query result" --> W W -. "3: store result back in Redis for next time" .-> R W --> C

应用/Web服务器先查Redis：命中则直接返回（快）；未命中则查数据库，拿到结果后再顺手写回Redis，供下次请求使用。

Layers of caching, from closest to the user to the database itself 从离用户最近到数据库本身的各层缓存

Layer 层级Example 示例Who manages it 谁管理
Browser cache 浏览器缓存Cached images/JS/CSS in the user's browserBrowser + web dev headers
CDN cache CDN缓存Cloudflare, Akamai, CloudFront edge cachingWeb/Infra team
App-level / distributed cache 应用级/分布式缓存Redis, MemcachedApp / Platform / DevOps team
DB engine's own internal cache 数据库引擎自身缓存Oracle Buffer Cache (SGA), PostgreSQL shared_buffers, MSSQL Buffer PoolDBA (this is the layer you already tune today)
Important distinction for DBAs: Redis is a completely separate, external cache that your application code chooses to check before ever reaching the database. It is not the same as the buffer cache/shared_buffers/buffer pool you already manage inside Oracle/PostgreSQL/MSSQL — that one caches raw data pages and is invisible to the app; Redis caches whole application-level results and is explicitly read/written by app code.
对DBA来说的关键区别：Redis是应用代码在"到达数据库之前"主动查询的一个完全独立、外部的缓存层。它与你现在管理的Oracle/PostgreSQL/MSSQL内部缓存（Buffer Cache/shared_buffers/Buffer Pool）不同——那些缓存的是原始数据页且对应用透明；而Redis缓存的是整份应用层结果，需要应用代码显式读写。

Why teams add Redis 为什么团队要用Redis

  • Cuts repeated, expensive DB reads — the same popular query result is served from memory instead of re-executed. 避免重复执行相同的高成本查询，直接从内存返回结果。
  • Microsecond-level response time (RAM) vs milliseconds for a disk-backed DB query. 内存级响应时间（微秒级），远快于依赖磁盘的数据库查询（毫秒级）。
  • Also commonly used for session storage, rate limiting, and pub/sub messaging — not just query caching. 除了查询缓存，也常用于会话存储、限流和发布/订阅消息。

Two "cold vs warm" patterns that look similar but aren't 两种看似相同、实则不同的"冷热"现象

You now have two different layers that both cause "first time slow, then fast" behavior — telling them apart matters for the team discussion:

Layer 层级Why first load is slower 为什么首次慢How to tell which one it is 如何判断
Riverbed (WAN optimization)Data-reduction library hasn't seen these bytes before. 数据缩减库尚未见过这些字节。Capture on the WAN leg — byte count is high on first load, low on repeat, even though the request/query itself is identical.

Talking-point script: "Slow-then-fast doesn't automatically mean Riverbed. If we see a DB query happen on the first request but not on the second, that's Redis cache warm-up, not WAN optimization — two different layers, two different owners."
会议话术参考："先慢后快不一定就是Riverbed的功劳。如果第一次请求时能看到数据库查询、第二次却没有，那是Redis缓存预热的效果，不是广域网优化——是两个不同的层，归属不同的团队。"

A common incident pattern: cache stampede 常见故障模式：缓存击穿/雪崩

When a popular cache key expires (TTL runs out), many concurrent requests can miss the cache at the same moment and all hit the database simultaneously — a sudden DB load spike that looks like "the database suddenly got slow" but is really a caching/expiry design issue.
当一个热门缓存键过期（TTL到期）时，大量并发请求可能同时未命中缓存，全部同时打到数据库——表现为"数据库突然变慢"，但根源其实是缓存过期策略的设计问题。

If the DB shows a sudden spike in identical queries at the same moment a cache TTL would have expired, ask the app/platform team about the Redis TTL and eviction policy (e.g. maxmemory-policy) before assuming a DB-side regression.
如果数据库在某个时间点突然出现大量相同查询，且该时间点恰好可能是某个缓存TTL到期的时刻，应先向应用/平台团队询问Redis的TTL与淘汰策略（如maxmemory-policy），而不是先假设是数据库本身的问题。

11. Latency & Error Diagnostics 延迟与故障诊断

FilterMeaning 含义
tcp.analysis.retransmissionPacket re-sent — network loss. 数据包重传，网络丢包。
tcp.analysis.duplicate_ackReceiver signaling missing data. 接收方提示数据缺失。
tcp.analysis.ack_rtt > 0.2Round-trip time over 200ms. 往返时延超过200毫秒。
tcp.analysis.zero_windowReceiver buffer full, sender must pause. 接收缓冲区已满。
tcp.flags.reset==1Connection forcibly closed. 连接被强制关闭。
tcp.analysis.flagsAny expert-flagged TCP problem. 任何专家诊断标记的TCP问题。
Rule of thumb: low ack_rtt but big gap before the response = server-side processing delay. High ack_rtt + retransmissions = network problem. If a load balancer or Riverbed is in the path, check which segment shows the delay before blaming the backend.
经验法则：ACK时延低但响应前有明显间隔 → 服务端处理延迟；ACK时延高且伴随重传 → 网络问题。若路径中有负载均衡器或Riverbed，先确认延迟出现在哪一段，再下结论是否是后端问题。

Do's and Don'ts before sharing a capture 分享抓包文件前的注意事项

✅ Do
  • Trim to the relevant window: File → Export Specified Packets (only the filtered/marked packets).
  • Note the capture start time and timezone in your message.
  • Confirm whether the payload is plaintext (HTTP/pgsql/tds/tns without TLS) before sending externally.
  • State clearly which segment you captured (client-side, LB-side, backend-side) if middleboxes are involved.
❌ Don't
  • Don't send a full, unfiltered production capture — it's large and may contain unrelated sensitive traffic.
  • Don't share captures containing plaintext credentials or customer data over open channels — check with security first.
  • Don't leave a capture running indefinitely on a production box — it fills disk and can add CPU overhead.
  • Don't call it "packet loss" from a single-segment capture if a Riverbed/LB split could explain the pattern.
抓包文件（.pcap/.pcapng）可能包含明文密码或客户数据，对外分享前务必先做脱敏处理，并与安全团队确认。

15. Practice Suggestions 练习建议

  1. Capture a browser visit to a plain HTTP site; find the GET request and response headers.
    抓取一次访问HTTP网站的过程，找到GET请求与响应头。
  2. Capture a login + query against MSSQL, PostgreSQL, and Oracle; identify each handshake (TDS Login7 / pgsql Startup / TNS Connect).
    分别抓取MSSQL、PostgreSQL、Oracle的登录与查询，识别各自的握手过程。
  3. If you have access to a load-balanced app, capture simultaneously on the client side and backend side; compare source IPs and timestamps.
    若有条件，在负载均衡应用的客户端侧与后端侧同时抓包，比较源IP与时间戳差异。
  4. If the app uses Redis, filter tcp.port==6379 and request the same page twice — compare the first (cache MISS + DB query) against the second (cache HIT only, no DB traffic).
    若应用使用了Redis，用tcp.port==6379过滤，连续请求同一页面两次，对比第一次（未命中+数据库查询）与第二次（仅命中，无数据库流量）的区别。
  5. Use Follow Stream to locate the SQL text or HTTP body in the payload.
    用Follow Stream找到载荷中的SQL文本或HTTP正文。
  6. Force a connection drop and locate the RST packet — note which side sent it.
    主动中断一次连接，找到RST包并确认由哪一方发出。
  7. Write up one finding using the 3-line template above and share it with a colleague — practice explaining it to a non-Wireshark user.
    用上面的三行模板写一条结论，分享给同事，练习向不懂Wireshark的人讲清楚。
